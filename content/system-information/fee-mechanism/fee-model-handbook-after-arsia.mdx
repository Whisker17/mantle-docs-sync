**Warning**

The content of this fee model is effective only after the Mantle Arsia upgrade. It is currently valid solely on **Mantle Sepolia**.


### Overviews

As a Rollup, Mantle has operated under a stable fee calculation model since the launch of Mantle v2 Tectonic. However, as the underlying technology has evolved, the original fee model is no longer fully compatible with current requirements. To address this, Mantle introduces a new Fee Model that modularizes transaction fees and aligns with the OP Stack’s Jovian upgrade, clearly distinguishing different sources of cost.

#### The Pre-Arsia Era

Before the Arsia upgrade, Mantle adopted a relatively coarse **"gas amplification"** strategy to handle L1 data costs. In this model, L1 costs were not explicitly separated; instead, they were implicitly embedded into L2 gas consumption. This manifested in the following ways:

* Gas amplification mechanism
  * Because Mantle uses **MNT** rather than **ETH** as its gas token, and the price difference between the two can be significant, Mantle’s gas values (e.g., `gasUsed`) differed by orders of magnitude from other L2s. This increased the adaptation burden for developers.
* Inaccurate estimation semantics
  * The legacy model converted L1 data costs and included them in the value returned by `eth_estimateGas`. As a result, L2 gas usage became tightly coupled with L1 fee market fluctuations. Even for identical transactions (e.g., a simple transfer), the estimated gas limit could vary significantly over time due to L1 fee volatility. This undermined the stability of gas limits as a measure of computational complexity and made accurate cost estimation difficult for developers.

#### The Post-Arsia Era

The current Mantle fee model adopts a **modular and decoupled design**. The total cost of a transaction is now clearly decomposed into three independent components, each corresponding to a distinct category of network resource consumption:

```rust
TotalFee = L2ExecutionFee + L1DataFee + OperatorFee
```

This model is aligned with the OP Stack Jovian upgrade and is specifically designed to support more complex and modular blockchain execution environments, such as ZK Rollups.


<Callout type="info">
For frontend developers, the `TotalFee` of the current transaction can be estimated by calling the `estimateTotalFee` method provided in the Mantle-Viem SDK.&#x20;

For backend developers, a similar `eth_estimateTotalFee` RPC is available in Geth, which directly returns the total fee without requiring manual aggregation of individual components.
</Callout>


### L2 Execution Fee

#### Design Rationale and Mechanism

The **L2 Execution Fee** is the most fundamental component of the fee model. It strictly follows Ethereum’s standard EVM gas accounting rules and directly corresponds to the computational resources and state storage resources consumed by the Mantle Sequencer when executing a transaction. Its pricing fully adheres to the EIP-1559 model and is jointly determined by the **Base Fee** and the **Priority Fee**.

</Callout>


#### Formula

With L1-related costs removed from gas estimation, the L2 execution fee becomes a direct function of gas usage and the EIP-1559 pricing parameters on Mantle L2. The fee calculation can therefore be expressed as:

```rust
// Calculate Total L2 Execution Fee
L2ExecutionFee = L2GasUsed * L2GasPrice
```

#### How to Retrieve the L2 Fee

Retrieving the L2 Fee follows the standard Ethereum development workflow. You can use the standard RPC interface `eth_estimateGas` to obtain the amount of gas required for the transaction, and then use `eth_gasPrice` to fetch the current gas price.


<Callout type="info">
Under Mantle’s current design, the value returned by `eth_estimateGas` reflects only pure L2 execution costs. Unlike implementations prior to Mantle Arsia, it no longer includes L1 data costs. Therefore, when calculating the total transaction fee, it is critical not to omit the other two fee components.
</Callout>


### L1 Data Fee

#### Design Rationale and Mechanism

As a Rollup, Mantle must publish transaction data to Ethereum mainnet in order to inherit its security guarantees. Since Ethereum data storage is expensive, Mantle introduces a data compression–based fee estimation mechanism.The core of the L1 fee model is a **linear regression–based cost estimator**. Mantle does not compute the exact compressed size for each transaction by running full compression, as doing so would introduce significant time overhead in the sequencing pipeline. Instead, following the OP Stack approach, Mantle first applies the FastLZ algorithm to obtain an approximate size, then uses a pre-trained linear regression model to approximate the final on-chain data size. Although not perfectly precise, this estimation is sufficiently accurate for pricing purposes. The final fee is computed by combining the fitted `BaseFeeScalar` and `BlobFeeScalar`, and then converting from ETH to MNT using the ETH/MNT `TokenRatio`.

#### Formula

Based on the estimation pipeline described above, the L1 fee can be expressed as a function of the estimated compressed data size, Ethereum L1 fee parameters, and the token conversion ratio. Formally, the L1 fee calculation is defined as follows:

```rust
// 1. Estimate compressed data size (Linear Regression)
EstimatedSize = COST_INTERCEPT + (COST_FASTLZ_COEF * _fastLzSize)

// 2. Compute weighted L1 fee (ETH)
WeightedL1Fee = (BaseFeeScalar * L1BaseFee * 16) + (BlobFeeScalar * L1BlobFee)

// 3. Compute final L1 data fee (MNT)
L1DataFee = EstimatedSize * WeightedL1Fee * TokenRatio
```

#### How to Retrieve the L1 Fee

Developers can query the on-chain `GasPriceOracle` contract and call the `getL1Fee(txBytes)` method to obtain an estimated value. To do so, you must provide the fully constructed, unsigned, and RLP-encoded transaction bytes as the input parameter in order to receive an L1 Fee estimate.


<Callout type="info">
Note that the value returned by this method is still denominated in ETH. To calculate the actual L1 Fee in Mantle’s native token (MNT), you must multiply it by the `TokenRatio`, which can also be retrieved from the `GasPriceOracle` contract.
</Callout>


Therefore:

```rust
L1 Fee = getL1Fee(txBytes) * TokenRatio
```

### Operator Fee

#### Design Rationale and Mechanism

The Operator Fee is designed to provide node operators with a sustainable revenue stream. It adopts a hybrid model of “fixed base + dynamic increment”: the fixed component covers baseline service costs, while the dynamic component scales linearly with transaction computational payload (gas used).

This makes network operational costs transparent and allows parameterized adjustment in response to changing network load.</td></tr></tbody></table>

#### Formula

Following this design, the Operator Fee is computed as the sum of a constant base fee and a gas-proportional component, as defined by the formula below:

```rust
OperatorFee = OperatorConstant + (OperatorScalar * 100 * L2GasUsed)
```

#### How to Retrieve the Operator Fee

Similar to the L1 Fee, the Operator Fee is also obtained via the `GasPriceOracle` contract. The contract exposes an interface called `getOperatorFee`. You must pass the L2 gas estimate obtained from `eth_estimateGas` as the input parameter. Based on the network’s currently configured scalars and constants, the contract will compute the Operator Fee that you are required to pay.
