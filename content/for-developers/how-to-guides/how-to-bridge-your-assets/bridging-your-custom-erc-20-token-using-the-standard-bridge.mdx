### When Do You Need a Custom L2 Token?

The standard `OptimismMintableERC20` implementation is designed to cover the most common bridging use cases. Tokens created using the factory contract are:

* Fully compatible with the Mantle Standard Bridge
* Simple, minimal, and non-upgradeable
* Suitable for most canonical L2 representations of L1 tokens

However, this standard implementation intentionally limits extensibility.If your project requires **additional logic**, such as:

* Upgradeability
* Pausing or emergency controls
* Blacklists or allowlists
* Custom hooks on deposit or withdrawal

You **must** use a custom L2 token implementation.This tutorial explains how to build a **custom, upgradeable ERC-20 token** that remains compatible with the Mantle Standard Bridge, while allowing you to safely extend functionality over time.


<Callout type="info">
Custom logic must not modify bridge-critical behavior.&#x20;

Incorrect implementations can break bridge accounting or invalidate security assumptions.
</Callout>


### Prerequisites

Before starting this tutorial, make sure that:

1. You already have an ERC-20 token deployed on Ethereum (L1)
2. Your token is **not** a fee-on-transfer or rebasing token
3. You have gas tokens on both L1/L2 networks. (For testnet, which are Ethereum Sepolia and Mantle Sepolia, check this to get testnet token)
4. You are comfortable deploying contracts using Remix

If you do not need custom functionality, consider following the tutorial on **Bridging Your Standard ERC-20 Token Using the Standard Bridge** instead.

### Deploy an Upgradeable L2 Token with Remix

In this section, you will deploy a **custom, upgradeable L2 ERC-20 token** that is compatible with the Mantle Standard Bridge.

Unlike the standard `OptimismMintableERC20`, this approach allows you to introduce additional functionality such as upgradeability, access control, or custom transfer logic, while still preserving correct bridge behavior.

This tutorial uses [**Remix**](https://remix.ethereum.org) to simplify deployment. You may adapt the same pattern to frameworks such as Hardhat or Foundry if preferred.


<Callout type="info">
This tutorial demonstrates one possible approach to creating a custom, upgradeable ERC-20 token for use with the Standard Bridge.

It may **not be suitable for all use cases**. Contract design choices such as upgrade patterns, permission models, and security assumptions should be carefully reviewed before deploying to production.

Developers are encouraged to rely on battle-tested implementations and to review official resources, such as the [OpenZeppelin Contracts documentation](https://docs.openzeppelin.com/contracts/5.x)
</Callout>


1. **Open Remix**

Navigate to [Remix](https://remix.ethereum.org/) in your browser.

Make sure your wallet is connected to **Mantle Sepolia** and that Remix is using the **Injected Provider** environment.

2. **Create a new contract file**

Create a new Solidity file, for example:

```solidity
MyUpgradeableL2Token.sol
```

Below is a **minimal upgradeable ERC-20 example** that:

* Implements the `IOptimismMintableERC20` interface
* Uses OpenZeppelin’s upgradeable contracts
* Preserves the required `mint` / `burn` semantics for the Standard Bridge
* Can be safely upgraded via a proxy

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import { Initializable } from "@openzeppelin/contracts-upgradeable/proxy/utils/Initializable.sol";
import { UUPSUpgradeable } from "@openzeppelin/contracts-upgradeable/proxy/utils/UUPSUpgradeable.sol";
import { ERC20Upgradeable } from "@openzeppelin/contracts-upgradeable/token/ERC20/ERC20Upgradeable.sol";
import { AccessControlUpgradeable } from "@openzeppelin/contracts-upgradeable/access/AccessControlUpgradeable.sol";

/// @notice Interface required by the Mantle / OP Standard Bridge
interface IOptimismMintableERC20 {
    function remoteToken() external view returns (address);
    function bridge() external view returns (address);
    function mint(address to, uint256 amount) external;
    function burn(address from, uint256 amount) external;
}

/// @title Upgradeable L2 ERC-20 compatible with the Mantle Standard Bridge
/// @notice This contract demonstrates how to build an upgradeable token
///         without breaking bridge security assumptions.
contract UpgradeableL2ERC20 is
    Initializable,
    ERC20Upgradeable,
    AccessControlUpgradeable,
    UUPSUpgradeable,
    IOptimismMintableERC20
{
    /// -----------------------------------------------------------------------
    /// Roles
    /// -----------------------------------------------------------------------

    bytes32 public constant BRIDGE_ROLE   = keccak256("BRIDGE_ROLE");
    bytes32 public constant UPGRADER_ROLE = keccak256("UPGRADER_ROLE");

    /// -----------------------------------------------------------------------
    /// Bridge-related storage (MUST remain stable across upgrades)
    /// -----------------------------------------------------------------------

    address private _bridge;
    address private _remoteToken;

    /// -----------------------------------------------------------------------
    /// Initializer
    /// -----------------------------------------------------------------------

    /// @notice Initializes the token. Replaces constructor for upgradeable contracts.
    /// @param bridge_ Address of the Mantle L2 Standard Bridge
    /// @param remoteToken_ Address of the corresponding L1 ERC-20 token
    /// @param name_ ERC-20 name
    /// @param symbol_ ERC-20 symbol
    /// @param upgrader Address that controls upgrades
    function initialize(
        address bridge_,
        address remoteToken_,
        string memory name_,
        string memory symbol_,
        address upgrader
    ) external initializer {
        require(bridge_ != address(0), "Invalid bridge");
        require(remoteToken_ != address(0), "Invalid remote token");

        __ERC20_init(name_, symbol_);
        __AccessControl_init();
        __UUPSUpgradeable_init();

        _bridge = bridge_;
        _remoteToken = remoteToken_;

        _grantRole(BRIDGE_ROLE, bridge_);
        _grantRole(UPGRADER_ROLE, upgrader);

        _setRoleAdmin(BRIDGE_ROLE, DEFAULT_ADMIN_ROLE);
        _setRoleAdmin(UPGRADER_ROLE, DEFAULT_ADMIN_ROLE);

        _grantRole(DEFAULT_ADMIN_ROLE, upgrader);
    }

    /// -----------------------------------------------------------------------
    /// IOptimismMintableERC20
    /// -----------------------------------------------------------------------

    function bridge() external view override returns (address) {
        return _bridge;
    }

    function remoteToken() external view override returns (address) {
        return _remoteToken;
    }

    /// @notice Mint tokens on L2 (called by the Standard Bridge on deposit)
    function mint(address to, uint256 amount) external override onlyRole(BRIDGE_ROLE) {
        _mint(to, amount);
    }

    /// @notice Burn tokens on L2 (called by the Standard Bridge on withdrawal)
    function burn(address from, uint256 amount) external override onlyRole(BRIDGE_ROLE) {
        _burn(from, amount);
    }

    /// -----------------------------------------------------------------------
    /// Upgrade Authorization
    /// -----------------------------------------------------------------------

    function _authorizeUpgrade(address newImplementation)
        internal
        override
        onlyRole(UPGRADER_ROLE)
    {}
}
```

3. **Deploy the proxy**

Using Remix:

1. Deploy the implementation contract
2. Deploy a **UUPS proxy** pointing to the implementation
3. Call `initialize` on the proxy with:
   1. `_bridge`: Mantle L2 Standard Bridge address
   2. `_remoteToken`: L1 ERC-20 token address
   3. `_name`, `_symbol`: token metadata


<Callout type="warning">
Always interact with the **proxy address**, not the implementation address.
</Callout>


Once deployed, this proxy address represents your **L2 bridged token**.

4. **Bridge tokens**

After deployment, you can bridge tokens from L1 to L2 using the Mantle Standard Bridge or [Mantle Viem](/for-developers/how-to-guides/how-to-use-mantle-viem).

At this point, your upgradeable token behaves identically to a standard bridged token from the user’s perspective.

### Add your token to the Mantle Token List

Mantle maintains an official token list that is used by infrastructure such as the Mantle Bridge UI and other ecosystem tools.

If you want your token to be discoverable and properly displayed across the Mantle ecosystem, you can submit it to the [Mantle token list](https://github.com/mantlenetworkio/mantle-token-lists) following the review and inclusion guidelines.
